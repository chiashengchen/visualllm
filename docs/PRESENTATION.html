Speech in, a talking face out — in under three seconds. mic → STT → LLM → TTS → photoreal avatar → browser

You speak; the system transcribes you, an LLM answers, the answer is spoken in a cloned voice, and a GPU renders a photoreal, lip-synced face saying it back — multi-turn, fully streaming over WebRTC. Everything heavy runs locally on one Windows box. The whole engineering story is one number: time-to-first-output.

Device specification

Everything heavy is local. The one hard constraint: a single GPU with ≥16 GB VRAM, because the TTS engine and the avatar renderer share that one card.

Hardware

1 NVIDIA GPU · ≥16 GB VRAMdev box: RTX 5060 Ti 16 GB · Blackwell sm_120 CosyVoice (vLLM) and MuseTalk share one card. Below ~16 GB they fight for VRAM → the avatar stalls or TTS fails to load.

The central constraint

One shared 16 GB GPU MuseTalk (Windows) and CosyVoice's vLLM (WSL) share ONE card. This single fact drives most of the system's hard decisions:
  • Load order: start CosyVoice before MuseTalk — vLLM needs the card mostly free at load.
  • Turn-start contention: both burst the GPU at once — the historic root of "mouth moves before the voice."
  • The only guaranteed cure for the residual ~0.5–1.5 s is a dedicated avatar GPU (working set ~4.8 GB).

Why the overlap matters The LLM's first sentence reaches TTS before the full answer exists, and TTS's first audio chunk reaches the avatar immediately. Chained naïvely — each stage waiting for the last — four heavy models in series would take 15–30 s per turn. Streaming overlap plus an LLM pre-warm collapse that to ~3 s.

A/V synchronization — steady vs live

steady · default · video-master Voice is buffered and released paced to the server's real rendered frames, so it waits when the render stalls and never drifts ahead — a synced start (the user's pick). Tradeoff: a long render stall briefly pauses the voice, then resumes clean.
live · audio-master · rejected Voice forwarded immediately so it can never freeze; lips best-effort (~0.75 s trail). Rejected by the user — voice leading the lips by 1–2 s is worse than an occasional pause.

Load-bearing: video_out_is_live = not avatar_sync_with_audio · one fps everywhere (server stride = client clock = video_out_framerate = MUSETALK_FPS) or A/V drift.

How the avatar posts frames

WebRTC and mobile decoders freeze on bursty input, so delivery is decoupled from rendering: a GPU worker produces frames in bursts; a pump emits them at a rock-steady rate.

GPURender worker
Buffers the incoming voice, runs UNet+VAE on fixed 8-frame segments (each ceil-sized to exactly cover its audio). Bursty — fast when free, stalls under contention.
push rendered RGB frames
bufferout_q · bounded queue (~600)
A smaller cap makes the render skip stale frames instead of letting the lips fall arbitrarily behind — MUSETALK_OUT_Q is the safe lag lever.

Emits exactly one frame every tick — never a burst. So the stream never gaps (a single skipped send is a visible freeze).

The reads

Parallel synthesis Sentence 2 「您想了解更多嗎？」 is synthesized in parallel (1.96 → 2.83 s) while sentence 1 already plays — so it costs TTFO nothing. That is the payoff of sentence streaming.
The two dominant costs = the two attacks Inside the pipeline the LLM first token (1.07 s, unpinned here) and the CosyVoice first chunk (0.875 s) dominate — exactly what §04 targets. Pinned to Groq, this turn lands nearer ~2.2 s.
The +1.99 s is a remote number Viewed over Tailscale, the last mile is network / jitter-buffer bound and swings with the link (0.94 s one run, 1.99 s another). A local viewer's last mile is far smaller (~0.15–0.8 s). The pipeline is not the lever there.

Measurement law (twice-proven): A/B on the live full stack only, and the user's live eye is the final gate — the offline probe has passed configs the eye then rejected.

Recorded with the same rigor as the wins, so they're never re-tried. Every one was measured, not guessed.

The through-line On a single shared GPU, most latency levers that win in isolation lose in the live system, and the human eye is the final gate the objective probes repeatedly fail. The two remaining levers, honestly: (1) shrink the TTS first-chunk further — input-splitting is near its floor, the compute side is capped by the zh-lipsync/RAS constraint; (2) a dedicated avatar GPU — the only guaranteed structural cure for the residual shared-GPU cost.
