The turn, as a clock

t = 0 is the instant the user stops speaking (the <8s stopwatch starts here). The shaded band is the TTFO window. Hover any bar or dot for the why. Notice the LLM, TTS and avatar lanes overlap — that is the whole trick. Hollow rings (◯) mark where a node receives input; filled dots (●) where a node emits output.

Node-to-node handoffs — when each node gets its input

All four arrivals are on the pipeline's single clock (relative to t0). Important: the +3.19s "out" is when the voice starts — the avatar forwards audio immediately (~7ms after the chunk) without waiting for a rendered frame. The lip-synced video is produced best-effort and decoupled: MuseTalk sustains real-time throughput (probe: 12.0 fps, ~83ms/frame, no freeze), but the intra-server render latency isn't logged, the lips can trail the voice under shared-GPU load, and the exact lip-vs-voice offset wasn't captured this run (the recorded mp4 had no video track). So: real-time throughput, not zero-latency lip-sync.

Why TTFO is small (it’s overlap, not a sum)

The pipeline streams — the user hears sentence 1 while everything else is still cooking.

TTFO 3.19s ≈ LLM first sentence (0.93s) + CosyVoice first‑chunk TTFB (2.25s)

The LLM finishes the entire 46-token answer at +1.07s, and TTS sentences 2&3 and the rest of the avatar render all happen after the first words are already playing. None of that adds to TTFO. If the stages ran serially you’d wait for the whole answer to be generated, then fully synthesized, then rendered. Instead the critical path is just first-sentence → first-audio, and the user is hearing a reply at 3.19s.

On this run, TTS first-chunk (2.25s) — not the LLM — was the dominant cost. That’s the lever to watch (warm CosyVoice / vLLM first-inference). The transpacific LLM hop was a modest 0.89s.
